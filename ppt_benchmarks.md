# ppt 컨펌 사례 벤치마크 (Approved Deck Benchmarks)

`/ppt` 스킬(`ppt.md`)의 외부 참조 파일. 컨펌된 덱의 기록으로, **복사용 템플릿이 아니라 보정용 품질 기준**이다. `ppt.md` Step 0에서 이 파일을 Read로 먼저 읽고, 새 덱이 컨펌되면 여기에 한 항목씩 추가한다. 항목은 짧게, 이식 가능한 품질 규칙 중심으로. (사례가 늘어도 `ppt.md` 본문은 가볍게 유지하려 분리해 둔 파일이다.)

> 공개용 비식별화: 회사·제품·인물·금액 등 고유 정보는 일반 표현으로 치환했다. 핵심 가치인 **구조(spine)·문체(copy)·서사(story)·레이아웃 기준**은 그대로 유지. 본인 환경에서 새 사례를 추가할 땐 고유명사를 적어도 되지만, 공유 레포라면 비식별화 상태를 유지한다.

## 컨펌 등재 규칙 (Confirmation Protocol)
사용자가 PPT/PDF를 명시적으로 컨펌하면, 다음 `/ppt` 작업 전에 그 최종 덱을 다시 열어 **이 파일에 한 항목**을 추가한다: 날짜·제목·문서 유형/독자·승인된 슬라이드 순서·문체/register 규칙·디자인 규칙·일반화하면 안 되는 맥락 요소. **리뷰 중인 산출물은 등재하지 않는다.** 일반화 가능한 레이아웃·정렬 교훈은 이 파일이 아니라 `ppt.md`의 "검증된 레이아웃·정렬 교훈" 섹션에 번호 항목으로 추가한다.

### 사내 해커톤 기획안 · Confirmed
- Type: general planning deck, 14 slides
- Role: 임원·운영자 독자를 위한 사내 이니셔티브 제안
- Approved spine: cover → agenda → one-page summary → overview → purpose/background → theme/mission → stats → overall schedule → day timetable → judging criteria → participant benefits → post-program operation → core message → end
- Copy standard: 담백한 비즈니스 명사구, 본문에 서술형 종결문 금지, 번역투 영어구 대신 자연스러운 한국어 헤더
- Story standard: 한 장이 하나의 역할; 일정 개요가 상세 타임테이블에 선행; 심사 기준은 발표/심사 맥락 직후; 혜택이 운영 이후 계획으로 연결
- Design standard: 브랜드 Navy/Sky, Pretendard, 플랫 도형, 넉넉한 여백, 절제된 footer 시스템
- Do not reuse blindly: 해커톤 특화 섹션·참가 혜택 문구·사내 이벤트 전제

### 신규사업 기획안 (인력 양성·매칭형) · Confirmed
- Type: business plan / new business strategy deck, 17 slides
- Role: 시장 진입·모델 설계·리스크·의사결정 조건을 검토하는 경영진 보고용 사업 제안
- Approved spine: cover → contents → one-page summary → core thesis → demand analysis → supply analysis → structural limits → key challenge → reconstructed model → model detail → differentiated assets → revenue model/business viability → roadmap → risks and mitigation → decision recommendation → team → end
- Copy standard: 정제된 한국어 비즈니스 문체; 감정적·유아적·구어·번역투 금지; 모호한 수입 용어를 의사결정 가능한 한국어 주장으로 재작성
- Story standard: 문제와 해법을 인접 배치; 구조적 한계 → 핵심 난제 → 재구성 모델; 리스크는 의사결정 권고 직전
- Layout standard: 레이아웃은 인접성이 아니라 의미를 따름. 좌우=비교, 상하=문제→해결; 의미 구조가 같은 장표는 같은 레이아웃 문법
- Emphasis standard: 해결방안·완화책·변화 후 상태·핵심 결론은 Navy 밴드·강조 컬럼·결론 화살표 밴드 등 절제된 장치로 시각 강조
- Do not reuse blindly: 도메인 특화 시장 수치, 특정 커뮤니티·채용 채널 자산, 지명된 후보군, 레퍼런스 기업명, 미검증 시장 규모

### 지사 신규업무 추진 기획안 · Confirmed
- Type: short operational proposal / HR work-plan deck, 12 slides
- Role: 초안 업무 메모를 간결한 실행 중심 기획안으로 재구성하는 사내 HR 제안
- Approved spine: cover → contents → core thesis → current work and expansion direction → operating model → program 1 → program 2 → program 3 → roadmap → risks and mitigation → success criteria → end
- Copy standard: 간결한 한국어 비즈니스 문체; 원 사실은 유지하되 메모투 표현을 장표의 의사결정 역할을 드러내는 헤더로 교체
- Story standard: 짧은 운영 제안서는 한 장 요약을 생략하고, 로드맵·리스크·성과 기준이 이미 다음 단계를 설명하면 저신호 권고 장표를 넣지 않음
- Layout standard: core thesis 단순화; 문제 행 + 단일 해결 밴드, 장식용 미니칩 금지. 가로 연결 timeline은 실제 프로세스 흐름에만. 독립 기준은 연결선 없는 분리 카드
- Alignment standard: 원형 번호 마커는 원 내부 중앙 + 자기 카드의 가로 중심에 정렬
- Do not reuse blindly: 특정 지사·구성원·프로그램명·사옥 세부·상담 서비스 가정 등 맥락 고유 요소

### 사내 벼룩마켓 데이 기획안 · Confirmed
- Type: internal program/event plan, 11 slides — 가볍고 캐주얼한 사내 이벤트 제안
- Role: 사내 점심시간 벼룩마켓 행사를 제안하는 가벼운 운영 기획안
- Approved spine (사용자 표준 흐름 적용): cover → 목차 → 행사 요약(stats, 강조 테두리) → 기획 배경(문제·니즈) → 추진 아이템·기대효과(상하) → 진행 일정(timeline) → 참여 흐름(process-flow) → 거래 품목 기준(two-col 비교) → 운영 방식·거래 원칙(label-row) → 운영 준비물(table) → end
- Narrative standard: 목차→개요/요약→배경(문제)→그래서 이걸 함(추진 아이템+기대효과)→How(일정·흐름·운영)→마무리. 기대 효과는 끝이 아니라 추진 아이템과 함께 앞쪽. "추진 아이템·기대효과" 라벨은 문서 주제에 맞게 치환.
- Copy standard: 밝고 캐주얼하되 명사형 종결. 줄글 대신 `· 리드 + 설명` 불릿. 입력에 없는 수치 날조 금지(예산 근거 없으면 생략·대체). **구어/비유 표현 금지**(예: "멍석"→"거래 환경"). 핵심 원칙은 하단 결론 밴드보다 **제목 밑 소제목 한 줄** 선호.
- Layout standard: 좌우=비교, 상하=정의/근거→핵심(핵심·결론·기대효과는 하단). 초반 써머리는 footer 위에서 끝나는 매끄러운 단일 Sky 라운드 테두리로 강조(라운드 코너와 어긋나는 사각 장식 금지). 라벨-행 장표의 하단 takeaway가 군더더기면 제거.
- Do not reuse blindly: 이벤트 고유 품목·금액 상한·결제/기부 방식 등 이벤트 고유 조건, 캐주얼 톤 강도

### 음성 전사 솔루션 도입 기획안 · Confirmed
- Type: AX/업무자동화 솔루션 도입 기획안 (general-brief 변형), 13 slides — 경영진·실무 의사결정자 대상 내부 제안
- Role: 음성 기반 업무(회의·인터뷰·개발·교육·글로벌)가 많은 조직에 AI 음성 전사 솔루션을 단기 PoC로 작게 도입 후 확대하는 흐름을 제안
- Approved spine: cover → 목차 → 도입 요약(2×2 카드, 매끄러운 단일 Sky 라운드 테두리) → 도입 배경·현재 문제(label-row+takeaway) → 솔루션 개요(상단 정의 + 하단 핵심기능 하이라이팅 패널, 3×2 그리드) → 활용 시나리오·기대효과(5카드, 불릿 줄바꿈) → 도입 효과 수치 전망(stats) → PoC 운영 설계(timeline, 설명을 패널 내부 노드 아래 배치) → PoC 검증 기준(table+takeaway) → 보안·개인정보 대응(label-row+takeaway) → 확대 로드맵(노드·설명·일정박스 동일 중심축 정렬) → 리스크·대응(table+takeaway) → end
- Copy standard: 사내 AX 제안 톤, 명사형 종결. 예상 수치는 추정치임을 캡션에 명시(예: "업계 사례 및 솔루션 성능 기준 추정치"). 솔루션 가정 맥락의 기대치이며 고객/시장 고유 수치 아님
- Story standard: PoC(검증) → 결과 분석 → 부서 확대 → 전사 안착의 단계 서사. 보안을 전담 장표로 두고, 리스크 매트릭스에는 보안 행을 중복하지 않고 연동 지연·수용도·ROI 등 별도 리스크로 채움
- Layout standard ("검증된 레이아웃·정렬 교훈"과 동일): 다중행 텍스트는 `\n` 한 run 금지·줄별 para 분리, label-row 행높이=takeaway 밴드 높이 통일, 써머리 테두리는 사각 강조바 없는 단일 라운드, 노드 플로우와 하단 박스는 같은 column center 정렬, 노드 지름 1.3"+ 권장, 표 행높이 0.66~0.74로 압축
- Do not reuse blindly: 솔루션 특화 기능명·지원 언어 수·정확도 수치 등 가정 수치, 특정 조직의 음성업무 시나리오, 추진팀·일정 등 맥락 고유 요소

### 온보딩 경험 개선 기획안 · Confirmed
- Type: HR 신규업무·운영 기획안 (hr-work-plan), 14 slides — 기존 온보딩 프로세스를 시점 기반 표준 여정으로 개선하는 내부 제안
- Role: 입사자 정보 습득·경험이 담당자/팀에 따라 좌우되는 문제를, 입사 전·당일·1주·1개월 4시점 표준 온보딩으로 재설계
- Approved spine: cover → 목차(대주제+소주제 그룹) → 핵심 요약(2×2, 카드마다 제목/불릿 가로 구분선) → 추진 배경·문제(label-row+takeaway) → 개선 방향(상단 정의 + 하단 3원칙 카드) → 4단계 여정(timeline, 균일 알약 뱃지) → 입사 전/당일/1주/1개월 phase 상세 4장(동일 numbered 레이아웃) → 멘토멘티·멘토링 채널(two-col) → 기대 효과(stats + 정성 takeaway) → 실행 로드맵(flow3 동일 중심축) → end
- Copy standard: 사내 HR 제안 톤, 명사형 종결. 예상 수치(예: 탐색시간↓·적용률·만족도)는 "도입 후 측정할 목표치, 효과 보장 아님"을 캡션에 명시. 정량(stats)과 정성(takeaway 밴드)을 분리
- Story standard: 현재 문제(기준선) → 개선 방향 → 여정 개요 → 시점별 상세 → 멘토링(상시 지원) → 기대효과 → 실행 로드맵. 여정 개요(timeline) vs 시점별 상세(phase 장)의 역할 분리, 멘토 매칭은 phase엔 이벤트로만 두고 제도 상세는 전담 장표가 책임
- Layout standard ("검증된 레이아웃·정렬 교훈" #8~#10): 써머리 카드는 소제목↔불릿 사이 가로 구분선 + 위 악센트바 생략 + 불릿 시작 y를 충분히 올림(넘침 방지), 알약 뱃지는 가변폭 금지·균일 고정폭으로 동일 크기, 동일 구조 연속 장표는 목차에서 대주제+소주제로 묶고 그룹 목차엔 행 구분선 생략
- Do not reuse blindly: 특정 메신저 사용 전제, 4시점(전/당일/1주/1개월) 구분, 30일 레터·멘토링 채널 등 이 조직 온보딩 고유 요소와 목표 수치

### 스토리텔링형 사내 해커톤 기획안 · Confirmed
- Type: 사내 프로그램·행사 기획안 (internal-program-plan), 14 slides — 회사 정체성을 구성원 경험으로 전환하는 스토리텔링형 해커톤
- Role: 생성형 AI로 서비스 프로토타입을 만드는 사내 해커톤을 운영·축제화하는 행사 제안 (행사명·세부 컨셉·세계관으로 스토리텔링)
- Approved spine: cover → 목차(대주제+소주제 그룹) → 행사 요약(2×2, 화면 가장자리 프레임) → [기획 개요: 기획 목적 / 운영 컨셉 / 운영 기준] → [운영 계획: 전체 일정(timeline) / 참가·모집(two-col) / 행사 진행(flow)] → [심사·수상: 심사 기준(stats) / 수상 구성(table)] → 예산안(two-col 비교) → 핵심 메시지(statement) → end
- Copy standard: 명사형 종결, 행사 스토리텔링(행사 네이밍/세부 컨셉/세계관 메타포)을 운영 컨셉 전용 장표로 부각. 심사 배점·예산 안·수상 부문 등 입력 수치 그대로, 창작 없음
- Story standard: 일정 개요(timeline) ↔ 당일 진행(flow)을 역할 분리해 중복 제거. 스토리텔링 컨셉은 한 장에 모아 부각
- Layout standard ("검증된 레이아웃·정렬 교훈" #10~#14): 목차·구성은 의미 친화도로 묶기(레이아웃 달라도 상위 주제면 그룹), 화살표+밴드는 결론·방안 전용·중립 보조정보는 화살표 없는 note_band, statement 가운데 짧은 바 생략, 페이지 통째 강조는 화면 가장자리 프레임, 알약 뱃지는 패널 안에서 폰트로 폭 맞춤
- Do not reuse blindly: 행사 네이밍·세계관 메타포·수상명·예산 금액·오프라인 공간 등 이 행사 고유 요소.

### 채용 프로세스 개선 기획안 · Confirmed
- Type: 운영 개선 보고/제안 (operating-report 결), 14 slides — 기존 채용 프로세스를 진단하고 리드타임을 단축하는 내부 개선 제안
- Role: 긴 리드타임·복잡한 면접·담당자 분산으로 인한 정보 누락 문제를 진단하고, 면접 통합·보상 전문화·책임 일원화로 채용 기간을 절반 이하로 단축
- Approved spine: cover → 목차(현황진단·개선설계·개선안 3그룹) → 핵심 요약(2×2, 화면 가장자리 프레임) → [현황 진단: 현재 프로세스(timeline) / 문제 진단(label-row+takeaway)] → [개선 설계: 개선 방향(3원칙 카드) / As-Is→To-Be(two-col 비교)] → [개선안: 면접 축약 / 처우 가속 / 책임 일원화(동일 numbered)] → 기대 효과(stats + 정성 takeaway) → 실행 로드맵(flow) → 핵심 메시지(statement) → end
- Copy standard: 사내 운영 개선 제안 톤, 명사형 종결. 입력의 기간 수치(2.5개월→1개월, 면접 3→1회 등)는 그대로 사용, 기대효과 수치는 "도입 후 측정할 목표·추정치, 보장 아님" 캡션 명시. 정량(stats)과 정성(takeaway)을 분리
- Story standard: 현황(as-is 기준선) → 문제 진단 → 개선 방향 → As-Is/To-Be 비교 → 세부 개선안 → 기대효과 → 로드맵. 현재 프로세스(timeline)와 비교(two-col)의 역할 분리
- Layout standard ("검증된 레이아웃·정렬 교훈" #10·#13·#14): 의미 친화 묶음 목차(레이아웃 달라도 상위 주제면 그룹: 현황진단/개선설계/개선안), 그룹 슬라이드 키커에 공통 접두(DIAGNOSIS·/REDESIGN·/ACTION·), 화면 가장자리 프레임 써머리, timeline 노드 가장자리 inset
- Do not reuse blindly: 특정 조직의 단계별 기간·담당자 수(5인)·보상 담당자 영입 전제 등 맥락 고유 요소와 목표 수치
